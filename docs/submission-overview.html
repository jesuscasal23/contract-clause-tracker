What it does

Upload, label, browse — one workspace

  1. Upload a contract (.txt / .md) The text is stored verbatim and segmented into sentences once, at upload — pysbd handles the legal-text traps (Inc., e.g., 99.9%, Section 3.2).
  2. Label clause-bearing sentences Click a sentence → an in-place popover → pick a clause type from a seeded taxonomy of seven. Labels render as colored highlights; a sidebar shows the live per-document clause summary.
  3. Browse the whole corpus Every document with its clause counts, plus search, filter-by-clause and group-by-clause — all in a persistent left panel, so switching documents never leaves the page.

The backend is deliberately minimal — one FastAPI app, three routers, four tables (documents, sentences, clause_types, annotations), one segmentation module. Interactive API docs live at /docs, and a fresh install seeds one sample contract so the app never opens empty.

Four tables, one invariant

Key decisions

The four decisions that shaped the design

“A clause is a single sentence” constrains the unit of annotation, not the definition of a clause

Most sentences carry no label; a few are tagged from a fixed taxonomy. The system's job — for the human now, for a model later — is to find the needle sentences.

Why it matters: the taxonomy doubles as the dashboard's filter vocabulary and a future model's fixed output space, and sentence rows become the unit that humans and the model share.

Data model: sentence rows that also carry character offsets

Sentences are materialized as rows (stable ids, simple UI, trivial dashboard queries), and every annotation copies its sentence's char_start / char_end into the verbatim raw_text. The invariant raw_text[start:end] == text is verified at upload and in tests.

Why it matters: if real clause extraction later needs free (multi- or sub-sentence) spans, that's a constraint relaxation and a mechanical migration — not a rewrite.

Manual and automatic labels share one pipeline

annotations carries source (user | model), status (confirmed | suggested | rejected) and confidence from day one. Auto-labeling becomes: insert suggested rows via the scaffolded POST /documents/{id}/auto-label; the UI already renders them as dashed highlights with Accept / Reject (a single PATCH).

Why it matters: human-in-the-loop review is exactly what legal teams need — and the accept/reject stream becomes a ready-made training signal. Rejected labels are kept, not deleted.

Reuse a token-driven design system instead of hand-rolling components

The UI is built on a Spartan-NG library (the Angular port of shadcn/ui) whose entire look is driven by CSS design tokens — the Legartis palette (orange #fa5b05, cool-slate neutrals, Geist, tight radii) was taken from the live site and applied in one token block. The workspace ships a Storybook workbench for the full component set.

Why it matters: accessible, consistent components for free, and the time saved went into the graded centerpiece — the labeling UX.

What's next

The automatic-labeling step, ready to land

POST /documents/{id}/auto-label is scaffolded as a documented stub: run a classifier over a document's sentences, insert source='model', status='suggested' annotations with a confidence, and the existing UI takes over. Model options, cheapest first: few-shot LLM classification per sentence; embeddings + nearest clause-type prototype; or a classifier fine-tuned on the accumulated confirmed labels.

Try it: clone the repo and run docker compose up --build → app on localhost:4200, API docs on localhost:8000/docs. Full detail in the README and docs/requirements.md.
